osm到OSRM的处理流程

注意:
这个是基于5.x版本的OSRM。

1.读取：

osrm-extract hlj.osm -p ../osrm-backend/profiles/car.lua
osrm-extract二进制文件解析解析原始的osm数据，将其存储为中间格式。

  • 设置：.lua文件
  • 输入：原始的OSM数据。
  • 输出：中等的OSRM格式。

解析OSM数据

第一步：读取原始的OSM数据，并让LUA脚本解析和过滤数据。一个线程读取原始OSM数据块，而另一个线程处理数据。数据首先使用STXXL向量存储在临时文件中。

示例数据

输入数据由Testbot配置文件处理.数据表示以下OSM节点和方式：
Node map:

d
a b c
e

Ways:

nodes highway oneway
abc primary
cd primary yes
ce river
de primary

解析节点

配置文件包含LUA脚本代码。每个节点作为一个ImportNode对象传递给LUA node_function（）。LUA检查OSM标记，并在ImportNode对象上设置成员变量，以指示节点是否是障碍，是否有信号惩罚等等。

当node_function（）返回时，ExtractorCallbacks :: nodeFunction（）将节点添加到存储在ExtractionContainer中的STXXL磁盘向量中：

index id
0 1 # d
1 2 # a
2 3 # b
3 4 # c
4 5 # e

节点ID是从输入文件复制而来的，并不一定是（现实中不是）连续的，或者从零开始。矢量中的节点顺序遵循输入文件中的顺序。

解析方法

PBFParser :: parseWay（）首先将所有方法存储在本地向量中，每种方式都包含它传递的节点列表：

index id node ids
0 6 2,3,4 # abc
1 7 4,1 # cd
2 8 4,5 # ce
3 9 1,5 # de

与节点一样，从输入文件读取id可以是任意的。

然后，每个方法都将作为ExtractionWay对象传递给Profiles LUA的way_function（）。LUA检查OSM标签，并在ExtractionWay对象上设置成员变量，以指示方式是单向还是最大速度等。

在我们的例子中，Testbot配置文件返回以下方式：

way direction
abc bidirectional
cd oneway
ce bidirectional
de bidirectional

ExtractorCallbacks :: wayFunction（）现在将路径分成节点和节点。

如果ExtractionWay在前进/后退方向上有不同的设置，则为每个段存储两个InternalExtractorEdges，每个标记为单向。如果两个方向的设置相同，则存储单边，标记为双向。

在我们的例子中，结果如下：

index start node id target node id direction
0 2 3 bidirectional
1 3 4 bidirectional
2 4 1 oneway
3 4 5 oneway
4 5 4 oneway
6 1 5 bidirectional

在我们的例子中，方法ce是一个“河流”，并且由于LUA脚本为每个方向返回不同的速度而被分离。

请注意，表中的节点ID是指OSM ID，而不是向量索引。边缘存储了一个保存在ExtractionContainers中的STXXL磁盘矢量。

编写中间格式

ExtractionContainers :: PrepareData（）现在进行各种内部排序和预处理：

对usedNodeID中的节点进行排序

在所有节点中排序节点
在wayStartEndVector中排序方式。
在限制向量中排序限制从，然后修复开始。
在限制中排序限制向量，然后修复结束。
将限制写入.restrictions文件。
将用过的节点写入.osrm文件中：

index id ...
0 1 # d
1 2 # a
2 3 # b
3 4 # c
4 5 # e
